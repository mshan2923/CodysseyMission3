# CodysseyMission3

| **비교 항목** | **Make** | **Zapier** |
| --- | --- | --- |
| **UI / UX** | 시각적 노드 기반의 워크플로우 편집기  | 리스트(단계별) 형태의 직관적인 인터페이스  |
| **설정 난이도** | 초기 학습 곡선은 있으나 세밀한 설계 가능  | 매우 직관적이고 설정이 간편함  |
| **연동 서비스 범위** | 광범위하며 고급 커넥터 기능 제공  | 매우 광범위하며 가장 많은 서비스와 연동 가능  |
| **무료 플랜 범위** | 약 1,000 Ops/월 (상대적으로 넉넉함)  | 약 100 Tasks/월 (상대적으로 제한적)  |
| **실행 로그 확인** | 단계별 데이터 상세 확인 가능  | Task History를 통한 실행 이력 확인  |
- **Make**
    - **장점**: 시각적인 노드 편집으로 복잡한 조건 분기(Router)를 설계하기 매우 유리함.
    - **단점**: 인터페이스가 복잡해 보여서 처음에 적응하는 시간이 필요함.
    - **적합 상황**: 조건 분기가 많은 복잡한 워크플로우를 구현하거나, 무료 플랜 내에서 더 많은 실행 횟수를 확보해야 할 때.
- **Zapier**
    - **장점**: UI가 단순해서 누구나 5분 안에 자동화를 시작할 수 있을 만큼 접근성이 높음.
    - **단점**: 복잡한 조건 분기가 들어가면 리스트 형태가 길어져 관리가 어려울 수 있음. 무료 플랜의 작업 제한(Task limit)이 타이트함.
    - **적합 상황**: 간단한 Trigger-Action 구조로 빠르게 자동화를 구축하고 싶을 때.

**반복 업무 정의**: 업무 효율을 위해 매일 확인해야 하는 중요 메일의 내용을 디스코드 채널로 실시간 알림을 받고, 구체적인 이력을 구글 시트에 아카이빙하는 작업.  
**선정 도구**: Zapier   
• 
**선정 이유**: 학습 초기 단계에서 가장 직관적인 UI를 제공하여 빠르게 워크플로우를 구현할 수 있고, 트리거와 액션 간의 연결 과정이 매우 명확하기 때문. 
**워크플로우 흐름**:

1. **Trigger (Gmail - New Email)**: 특정 라벨이 붙거나 중요 키워드가 포함된 새 메일 수신 시 작동.  
2. **Filter (조건 분기)**: 중요 메일로 분류된 건만 선별하여 불필요한 알림 차단.  
3. **Action 1 (Discord - Send Channel Message)**: 디스코드 포럼(채널)에 메일 제목과 요약 내용을 즉시 전송.  
4. **Action 2 (Google Sheets - Create Spreadsheet Row)**: 추후 이력 관리를 위해 메일 수신 시간, 발신자, 제목을 구글 시트의 새로운 행으로 자동 저장.

- 두 도구 간 워크플로우 동일성 유지를 위한 설계 기준
Make와 Zapier는 데이터 처리 아키텍처와 인터페이스 구조가 완전히 다르므로, 두 환경에서 완벽히 동일한 결과값을 도출하기 위해 다음과 같은 엄격한 설계 기준을 적용함.

데이터 모델 데이터 흐름 표준화: Gmail 트리거에서 추출하는 객체(발신자 메일 주소, 수신 시간, 본문 텍스트)의 변수명을 상호 매핑하여 동일한 데이터 소스를 사용하도록 규격화함.

필터 룰 매칭: Zapier의 내장 필터(Filter by Zapier) 조건과 Make의 노드 간 필터 링크(Filter Setup) 조건을 완전히 일치시킴. (예: Subject 문자열 내에 특정 키워드가 Contains 상태인지 여부)

포맷 일관성: Discord로 전송되는 페이로드(Payload) 마크다운 문법과 Google Sheets에 적재되는 타임스탬프 포맷(YYYY-MM-DD HH:mm:ss)을 두 툴 모두 동일한 포맷 함수로 전처리하도록 라우팅함.

- 개념적 정의: Trigger와 Action의 차이 
노코드 파이프라인의 핵심 구조인 Trigger와 Action의 개념적 메커니즘과 동작 예시는 다음과 같음.

Trigger (트리거): 워크플로우의 실행을 촉발하는 '이벤트 발생 지점'임. 웹훅(Webhook)이나 폴링(Polling) 방식으로 외부 서비스의 상태 변화를 감지하여 파이프라인에 데이터를 주입하는 역할을 수행함.

예시: Gmail에 새로운 중요 메일이 수신되는 순간, 메일의 메타데이터를 가지고 자동화 시스템을 깨우는 행위.

Action (액션): 트리거된 데이터를 바탕으로 시스템이 실제로 수행하는 '기능적 연산 및 명령'임.

예시: 트리거가 넘겨준 메일 제목과 요약본을 디스코드 API를 호출해 채널 메시지로 발송(Action 1)하거나, 구글 시트 API를 통해 데이터베이스 행에 추가(Action 2)하는 행위.

-  비교 항목 연계를 통한 특정 도구(Zapier) 선정 이유 
최종 연동 도구로 Zapier를 선정한 이유는 단일한 UI 편의성을 넘어, 앞서 분석한 비교 항목 3가지(UI/UX, 설정 난이도, 연동 서비스 범위)를 교차 검증한 결론임.

UI/UX 및 설정 난이도 연계: 초기 프로토타이핑 단계에서는 복잡한 분기보다 빠른 검증이 필수적임. Zapier의 '리스트 형태 인터페이스'는 트리거와 액션의 흐름을 직관적인 탑다운(Top-down) 방식으로 보여주기 때문에 설정 난이도가 낮아 휴먼 에러를 최소화함.

연동 서비스 범위 연계: 향후 디스코드나 구글 시트 외에 사내 타 협업 툴(예: Slack, Notion, Jira 등)로의 확장성을 고려했을 때, 현존 노코드 툴 중 가장 광범위한 연동 에코시스템을 가진 Zapier를 선택하는 것이 장기적 유지보수 관점에서 비용을 절감하는 최적의 선택임.

- 심층 인터뷰 및 아키텍처 확장 전략
A. 트리거 불안정 시 모니터링 및 대체 경로 설계
모니터링: API 레이트 리밋(Rate Limit)이나 네트워크 타임아웃으로 인해 트리거가 유실되는 문제를 방지하기 위해, Zapier의 내장 기능인 'Alerts' 및 'Task History' 로그 알림을 활성화하여 에러 발생 시 즉각적인 디스코드 개발자 채널 피드백 시스템을 구축함.

재시도 전략: 일시적인 네트워크 오류에 대응하기 위해 무조건 3회 자동 재시도(Autoretry) 옵션을 적용함.

대체 경로(Fallback): 지속적인 API Fail 발생 시 데이터 유실을 막기 위해, 에러 캐치 노드를 설계하여 실패한 원본 메일 데이터 데이터를 임시 서브 데이터베이스(Data Store)나 백업 시트로 격리(Dead Letter Queue 방식)한 뒤 수동으로 일괄 재처리할 수 있는 대체 파이프라인을 구성함.

B. 조건 분기 증가 시 워크플로우 구조화 및 모듈화 (평가기준 14)
메일 종류나 부서별 알림 조건 등 분기가 5개 이상으로 증가할 경우, 하나의 대형 워크플로우에 모든 리스트를 붙이면 유지보수가 불가능해짐.

모듈화 전략: 이를 해결하기 위해 하나의 메인 트리거 숍(Shop)을 두고, Webhook이나 내부 서브 트리거 기능을 활용하여 개별 조건에 맞는 '마이크로 워크플로우(Sub-Zap)'로 데이터를 분기 처리하는 허브 앤 스포크(Hub-and-Spoke) 구조로 라우팅을 설계함. 이를 통해 특정 알림 조건에 오류가 발생해도 전체 파이프라인이 마비되지 않도록 결합도를 낮춤(Decoupling).

C. 노코드 자동화의 한계 및 코딩 기반 확장 아이디어 (평가기준 15)
노코드 파이프라인의 명확한 한계:

대용량 텍스트 파싱 및 복잡한 문자열 정규식(Regex) 처리의 제약.

데이터 포맷 변경이나 복잡한 2차원 배열 데이터의 루프(Loop) 연산 시 노코드 모듈 수 급증으로 인한 비용(Task 비용) 폭발 위험.

코딩 기반 확장 아이디어 (C# 및 AI 네이티브 연계):

Zapier 내부에 'Code by Zapier' (Python/Javascript) 스크립트 노드를 중간에 삽입하여 복잡한 메일 본문 데이터 필터링 연산을 원스톱으로 처리함.

한계가 더 깊어질 경우, 서버리스 환경(AWS Lambda 또는 Azure Functions)에 C# 기반의 커스텀 웹훅 수신기를 빌드하고, AI API(LLM)를 연동하여 메일 내용을 완전 자동 요약 및 분류한 뒤 가공된 클린 데이터만 최종 액션 툴로 던져주는 '하이브리드 코드-노코드 자동화 아키텍처'로 확장할 계획임.
